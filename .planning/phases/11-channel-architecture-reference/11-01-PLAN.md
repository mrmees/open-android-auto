## Fragmentation

Explain FrameType-driven reassembly:
- BULK = unfragmented (complete message in one frame)
- FIRST = first fragment, includes total message size in extended header
- MIDDLE = continuation fragment
- LAST = final fragment, triggers reassembly
- Fragments on a single channel MUST arrive in order; channels cannot interleave fragments.
Cite: MessageInStream.cpp:134-153 (resolve logic), MessageInStream.cpp:68-77 (interleave check).

## Service Discovery and Binding

This addresses ARCH-02 specifically.

1. **SDP exchange sequence ASCII art:**
   ```
   Phone                                    Head Unit
     |                                         |
     |   <--- ServiceDiscoveryRequest ---      |
     |   --- ServiceDiscoveryResponse --->     |
     |                                         |
     |   (for each needed channel:)            |
     |   <--- ChannelOpenRequest ---           |
     |   --- ChannelOpenResponse --->          |
     |                                         |
   ```
   Clarify: The HU sends the ServiceDiscoveryRequest; the phone responds with its own ServiceDiscoveryResponse. Both sides describe their capabilities. The HU then opens channels based on the negotiated set.

2. **SDP descriptors:** Each service in the SDP response is a `ChannelDescriptor` with a `channel_kind` oneof field (e.g., `av_channel`, `input_channel`, `sensor_channel`). The channel_kind determines the channel type and carries type-specific configuration.

3. **Service ID assignment:** The SDP assigns `channel_id` values dynamically per-session. The GAL service type is the stable identifier; wire channel_id is ephemeral. A given service (e.g., media audio) might be channel_id=4 in one session and channel_id=7 in another.

4. **Channel lifecycle state machine ASCII art:**
   ```
   CLOSED --> OPEN (ChannelOpenRequest/Response)
     --> SETUP (channel-specific config exchange)
     --> ACTIVE (data flow)
     --> CLOSED (ChannelCloseNotification or ByeBye)
   ```
   Cite: protocol-overview.md session lifecycle content (corrected), aasdk ControlServiceChannel.

~50-60 lines.

## Capability Negotiation

This addresses ARCH-03. Scoped to 3 concrete evidenced examples -- NOT an exhaustive catalog.

1. **Version Exchange:** First messages after TLS: VERSION_REQUEST (msg ID 1) / VERSION_RESPONSE (msg ID 2) on channel 0. Establishes protocol version (v1.7 observed). Gate for all subsequent communication.
   Cite: aasdk ControlMessageIdsEnum.proto, control channel verification report.

2. **Video Resolution Negotiation:** The SDP `av_channel` descriptor for MAIN display lists supported resolutions (e.g., VW offers 1920x1080 / 1280x720 / 800x480; DHU offers 720p only). The phone selects from the HU's offered set. Resolution change requires full ByeBye->reconnect -- no mid-session renegotiation.
   Cite: sdp-values.json, CLAUDE.md memory (video/DPI session).

3. **Material You Theming (ColorSchemeSupport):** AVChannel field 9 in SDP. Values: 0=basic (no theming), 2=Material You v2, 3=Material You v3. When set to 2+ on the MAIN display channel, the phone sends 16 ARGB color tokens on palette change. Phone-side gates: `HeroFeature__theming_enabled`, `HERO_THEMING` feature set, `setThemingUpdatesEnabled`.
   Cite: CLAUDE.md memory (Video Channel / DPI / Theming section), APK 16.2 analysis.

Close with: "For channel-specific negotiation details, see the individual [channel docs](.). For per-message field tables and msg IDs, see [protocol-reference.md](../protocol-reference.md)."
